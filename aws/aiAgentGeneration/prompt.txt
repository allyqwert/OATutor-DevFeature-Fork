VISUAL GRAMMAR (domain-neutral rules — apply to every visual):
- **Directed quantities → `arrow`, not `line`.** Any element that represents a direction, change, flow, gradient, displacement, derivative, vector, or any "from → to" relationship MUST use the `arrow` element type. Use `line` only for undirected segments (edges of shapes, axes, reference grids, chords, etc.).
- **Label every meaningful element, and make the label track.** For every arrow, distinguished point, region, or curve the student needs to reason about, include a `text` element whose `text` matches the symbol in your formulas (e.g. label an arrow that uses slider `F1` as `F_1`). The label MUST follow the element as sliders change — either (a) set `attach: { to: "<element-id>", side: "right", offsetX: 6 }` on the text element so it anchors to the target's bounding box and moves with it, or (b) reference the same slider-driven expressions in the label's `x` / `y` that drive the target. Do NOT hard-code label coordinates next to a moving shape; it will freeze in place.
- **Make conceptually-paired elements visually parallel.** If two elements represent the same kind of quantity, use the same `strokeWidth`, comparable lengths, and complementary positions. Contrast (color, thickness) should carry meaning, not be arbitrary.
- **Ensure sliders produce visible motion.** Every slider MUST drive at least one element's position, length, or size through an `=expression` (e.g. `x: "=d1"`, `width: "=F1*0.1"`, `fn: "=x*x + a"`). Referencing a slider only in a `text` label like `"d_1 = {{d1}}"` or inside a formula's display `latex` does NOT count — the label updates but nothing in the scene moves, which defeats the point of having a slider. The server rejects plans whose sliders do not wire into geometry.
- **Use the canvas.** Place elements so they occupy a meaningful fraction of the canvas at the default slider values. Avoid clustering everything in a corner. The renderer auto-fits each axis to the slider range referenced on that axis, so anchoring arrows at the origin `(0, 0)` and ending them at `(=varX, 0)` / `(0, =varY)` / `(=varX, =varY)` gives a clean layout.
- **Curves use the `function` element, not dense polylines.** To plot `y = f(x)`, emit `{ type: "function", fn: "=<expression in x>", xMin: "=...", xMax: "=..." }`. For parametric shapes (circles, ellipses, lissajous, rotating vectors over time), emit `{ type: "function", fnX: "=<x(t)>", fnY: "=<y(t)>", tMin: 0, tMax: "=2*PI" }`. Inside `fn`/`fnX`/`fnY` the independent variable is always the literal name `x` (single-var) or `t` (parametric); slider IDs may appear alongside. Do not approximate curves with hand-built `polyline` points.
